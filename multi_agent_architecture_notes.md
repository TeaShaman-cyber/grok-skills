# Multi-Agent Architecture Notes (Grok)

**Дата:** 2026-06-17 (имя чата: Каргат) — добавлен раздел про известные проблемы GitHub Connector + текущий статус (коннектор отвалился)
**Контекст:** Развитие системы, где Grok выступает главным агентом, а другие агенты (Copilot, другие чаты, будущие) — как вспомогательные консультанты через общие skills и коннекторы.

---

## Основная идея

- **Главный агент (Grok)** остаётся координатором и финальным ответственным.
- **Вспомогательные агенты** (Copilot, другие LLM-чаты, специализированные агенты) привлекаются для получения альтернативных точек зрения.
- **Наши skills и модули** (`actions.md`, `git-flow.md` и будущие) — общий контекст и "язык".
- **Коннекторы** (GitHub, Notion, Google Drive, Linear и т.д.) — **асинхронная общая память** и рабочая область между агентами.
- Каждый агент может иметь свою "оптику", стиль и предпочтения.

Цель: улучшить качество решений через множественные перспективы, не теряя контроля и coherence.

---

## Ключевые паттерны мульти-агентных систем (2025–2026)

### 1. Supervisor / Orchestrator Pattern (рекомендуется как стартовый)

- Один центральный агент (Supervisor) принимает задачу, планирует, делегирует подзадачи специализированным агентам и агрегирует результат.
- Самый стабильный и контролируемый паттерн для production.
- Хорошо подходит под наше видение (Grok = Supervisor).

**Плюсы:**
- Чёткий контроль
- Легко добавлять observability и governance
- Меньше хаоса, чем в peer-to-peer

**Минусы:**
- Supervisor может стать bottleneck'ом

### 2. Hierarchical Pattern

- Несколько уровней: высокоуровневые агенты делегируют более низкоуровневым.
- Полезно, когда есть сложная иерархия задач.

### 3. Pipeline / Sequential Pattern

- Агенты работают последовательно (Research → Draft → Review → Finalize).
- Хорошо для workflow-ориентированных задач.

### 4. Debate / Multi-Perspective Pattern

- Несколько агентов с разными "личностями" или экспертизами обсуждают проблему.
- Координатор выбирает лучшее решение или синтезирует.
- Полезно для архитектурных и стратегических решений.

### 5. Network / Peer-to-Peer (не рекомендуется как основной)

- Агенты общаются напрямую друг с другом.
- Может быть хаотично и трудноуправляемо.

**Рекомендация на старте:** Supervisor + 2–4 специализированных агента (включая Copilot как одного из них).

|---------------|------------------------------------------|------------------------------------|---------------|

- Не обязательно сразу брать тяжёлый фреймворк.

- Позже, при необходимости, можно добавить LangGraph-подобный слой.

## Ключевые принципы реализации (для нашего случая)

1. **Grok как Supervisor**
   - Я принимаю финальное решение.
   - Я решаю, когда и кого привлекать как консультанта.

2. **Skills и модули как общий контекст**
   - Перед обращением к вспомогательному агенту (Copilot) я могу "подсунуть" релевантные части наших модулей.
   - Это обеспечивает consistency и alignment с нашим подходом (Mem Palace, диагностика, практичность).

3. **Коннекторы как асинхронная общая память**
   - GitHub, Notion, Google Drive и т.д. могут выступать как места, где агенты оставляют артефакты, обсуждения, результаты.
   - Это позволяет асинхронную работу между разными чатами/агентами.

4. **Тихий режим по умолчанию**
   - Привлечение второго мнения происходит тихо.
   - Только в режиме отладки или по явному запросу показываем процесс.

5. **Специализация агентов**
   - Каждый привлекаемый агент (включая Copilot) должен иметь чёткую роль/оптику.
   - Не стоит делать универсальных "всезнаек".

6. **Именование и роли агентов (для снижения путаницы)**
   - В мульти-агентной работе полезно давать агентам/чатам рабочие имена или роли.
   - Это помогает отслеживать, кто что сказал и кто за что отвечает.
   - Пример текущего распределения ролей:
     - **Каргат (Системный координатор + Арбитектор навыков)** (этот чат): отслеживает целостность системы, формализует договорённости, анализирует правила и риски, ведёт артефакты.
     - **Натурфилософ + Исследователь** (другой чат): хорошо работает с большими идеями, метафорами, глубокими смысловыми связями и философской стороной ТИД.
     - **Главный исследователь и постановщик задач** (Семён): задаёт направление, принимает финальные решения, удерживает общий замысел Mem Palace и ТИД.
   - Роли не жёсткие и могут пересматриваться по мере развития системы.

7. **Observability и контроль**
   - Важно отслеживать, какие агенты привлекались, какие контексты передавались и какие решения были приняты.

### Правило работы с репозиторием `grok-skills`

Все агенты/чаты при работе с репозиторием `grok-skills` **обязаны** использовать MCP инструменты (`github___create_or_update_file`, `github___get_file_contents` и т.д.).

**Запрещено** имитировать работу с git через команды (`git add`, `git commit`, `git push` и т.п.) в терминале. Такая имитация не приводит к реальным изменениям в репозитории и нарушает синхронизацию артефактов между агентами.

## Потенциальные риски

- **Context dilution** — слишком много контекста от разных агентов может размывать фокус.
- **Increased latency and cost** — каждое обращение к дополнительному агенту стоит времени и денег.
- **Inconsistent quality** — разные агенты могут давать противоречивые рекомендации.
- **Loss of coherence** — если не держать сильный Supervisor, система может "развалиться".

**Меры противодействия:**
- Сильная роль Supervisor (Grok).
- Чёткие роли у вспомогательных агентов.
- Использование общих модулей как "якоря" контекста.
- Явный анализ и синтез нескольких точек зрения перед финальным ответом.

## Следующие шаги (предложение)

1. Начать с **Copilot как первого вспомогательного консультанта** через наши модули (`actions.md`, `git-flow.md`).
2. Зафиксировать простой протокол: "Когда привлекать второе мнение" и "Как передавать контекст".
3. Позже добавить других агентов (другие чаты Grok с разными инструкциями) и использовать коннекторы как общую память.
4. При необходимости ввести более формальную структуру (например, через LangGraph-подобный подход или явные "Agent Cards").

**Правила работы с общими артефактами:**
- Изменения вносятся через MCP инструменты (`github___create_or_update_file` с получением SHA).
- При значимых изменениях рекомендуется создать GitHub Issue или Discussion для согласования.
- В артефактах фиксируются **итоговые договорённости**, а не процесс обсуждения.

## Известные проблемы GitHub Connector (по состоянию на июнь 2026)

GitHub connector через официальный MCP имеет ряд известных проблем, которые влияют на надёжность работы:

### Основные проблемы:
- **Нестабильность write-операций** (commit / push / создание PR)
  - Частые intermittent failures с ошибками "API/tree state errors", JSON parsing issues и silent failures.
  - Операции записи значительно менее стабильны, чем чтение.
- **Проблемы с аутентификацией и токенами**
  - Intermittent auth/token refresh issues.
  - Коннектор может отваливаться посреди активной сессии.
  - Рекомендуемое решение: полное отключение и переподключение в Settings → Connectors, либо отзыв доступа в GitHub и повторная авторизация.
- **Проблемы с большими diff'ами**
  - При крупных изменениях коннектор иногда заменяет реальный код плейсхолдерами.
- **Общая ненадёжность write-flow**
  - Даже при сообщении "успешно запушено" часто требуется явно проверять результат (git log, commit SHA).

Из-за этих проблем **нельзя** делать использование GitHub Connector / MCP обязательным условием работы со скилами и артефактами. Необходимо сохранять альтернативные пути (локальные файлы + передача через Семёна).
